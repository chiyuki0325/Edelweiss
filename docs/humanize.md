# Naturalness Guidelines：如何让 bot 更像人

本文档记录了 Cahciua bot 消息自然度优化的调研过程、数据分析结果和最终写入 system prompt 的 guideline 设计依据。

## 1. 方法论

### 1.1 数据库消息分析

从 SQLite 数据库的 `events` 表中提取消息，用 `is_self_sent` 字段区分人类消息和 bot 消息。

过滤条件：
- 仅 `type = 'message'` 且 `text` 非空
- 长度 ≤ 200 字符（排除粘贴/转发等长文本噪声）
- 排除 `forward_info` 不为空的转发消息

最终样本量：
- 人类消息：32,274 条
- Bot 消息：3,984 条

对两组消息进行了 10 个维度的统计对比，并抽样原文做定性观察。

### 1.2 外部项目调研

并行调研了三个开源 IM bot 项目的 prompt 设计：
- **AstrBot**（AstrBotDevs/AstrBot）
- **MaiBot**（Mai-with-u/MaiBot）
- **proj-airi**（moeru-ai/airi）

通过 `gh api` 读取 GitHub 源码，重点关注 system prompt 中与"像人类一样说话"相关的指令设计。

## 2. 数据分析结果

### 2.1 消息长度分布

| 指标 | 人类 | Bot |
|------|------|-----|
| 中位数 | 12 字符 | 29 字符 |
| 平均值 | 20.4 字符 | 44.3 字符 |
| P25 | 6 | 13 |
| P75 | 23 | 62 |
| P90 | 45 | 106 |

长度分桶：

| 区间 | 人类 | Bot |
|------|------|-----|
| 1-10 字符 | 44.1% | 18.8% |
| 11-20 | 26.6% | 19.0% |
| 21-30 | 12.3% | 13.8% |
| 31-50 | 8.4% | 17.0% |
| 51-80 | 4.5% | 14.4% |
| 81-120 | 2.4% | 9.7% |
| 121-200 | 1.6% | 7.4% |

**结论：** Bot 消息显著偏长。人类近半数消息在 10 字以内，而 bot 在 10 字以内的只有不到五分之一。Bot 在 50+ 字符区间的占比远高于人类。

### 2.2 标点与格式

| 特征 | 人类 | Bot | 倍数 |
|------|------|-----|------|
| 无末尾标点 | 87.5% | 80.9% | — |
| 以句号结尾 | 5.1% | 14.0% | 2.7× |
| 以问号结尾 | 3.5% | 4.2% | 1.2× |
| 以感叹号结尾 | 1.2% | 0.4% | 0.3× |
| 以省略号结尾 | 2.7% | 0.5% | 0.2× |
| 含 emoji | 4.7% | 14.9% | 3.2× |
| 含 markdown | 0.1% | 0.0% | — |
| 大写字母开头 | 3.8% | 6.8% | 1.8× |
| 多句（含句间标点+后续文本） | 2.3% | 6.8% | 3.0× |
| 含换行 | 5.9% | 9.5% | 1.6× |

**结论：**
- Bot 过度使用句号（2.7 倍），人类在聊天中几乎不加句号
- Bot 使用 emoji 频率是人类的 3.2 倍
- Bot 发多句消息的频率是人类的 3 倍
- 补充观察：人类的轻停顿经常直接交给空格或句式本身，而不是显式补一个逗号或句号

### 2.3 句子复杂度

| 特征 | 人类 | Bot | 倍数 |
|------|------|-----|------|
| 逗号密集（<100字中≥3个逗号） | 1.8% | 5.9% | 3.3× |
| 含冒号 | 3.8% | 9.1% | 2.4× |
| 含破折号（——/—） | 2.4% | 16.7% | **7.0×** |
| 含括号 | 7.1% | 17.2% | 2.4× |
| 列表式开头（数字/bullet） | 1.4% | 2.3% | 1.6× |
| 过度礼貌用语 | 0.2% | 0.3% | 1.5× |

**结论：** 破折号是最显著的 bot 特征（7 倍差异）。括号和冒号也明显过度使用。

### 2.4 回复行为

| 特征 | 人类 | Bot |
|------|------|-----|
| 消息是回复（reply） | 45.8% | 92.1% |

**结论：** Bot 几乎总是以回复形式发送消息，这与其触发机制有关（被 @ 或被回复时触发），属于正常行为。

### 2.5 语言模式

| 特征 | 人类 | Bot |
|------|------|-----|
| 纯中文 | 54.9% | 48.7% |
| 纯英文 | 6.9% | 0.6% |
| 中英混合 | 36.0% | 50.2% |
| 含 lol/哈哈 | 0.4% | 0.7% |
| 含 haha/哈哈哈 | 0.1% | 0.1% |
| 含 www/草 | 1.3% | 0.3% |
| 单字消息 | 2.5% | 0.3% |
| 纯标点/符号 | 2.0% | 0.4% |
| 重复字符（4+连续） | 0.6% | 0.3% |

**结论：**
- Bot 几乎不发纯英文消息（0.6% vs 6.9%），中英混合比例偏高
- 人类更多发送超短消息（单字、纯标点如"？"）
- 人类更多使用 www/草 等中文互联网原生表达

### 2.6 连续消息（burst）行为

| 特征 | 人类 | Bot |
|------|------|-----|
| Burst 中的消息占比 | 37.5% | 65.9% |

**结论：** Bot 有更多消息处于连续发送状态（60 秒内同一发送者的连续消息），这与 bot 在一轮中发送多条 `send_message` 有关。

### 2.7 特定词频

| 特征 | 人类 | Bot | 倍数 |
|------|------|-----|------|
| 确实/的确/indeed | 1.9% | 7.0% | **3.7×** |
| 其实/actually | 1.2% | 1.4% | 1.2× |
| 感觉/I feel | 1.9% | 1.3% | 0.7× |
| 啊/呢/吧（句末语气词） | 3.2% | 1.2% | **0.4×** |
| 吗/么（疑问） | 4.6% | 3.3% | 0.7× |
| x了 结尾 | 9.3% | 8.3% | 0.9× |
| 省略号 | 4.5% | 7.1% | 1.6× |
| 波浪号 ~ | 0.9% | 1.5% | 1.7× |

**结论：**
- "确实"是 bot 最显著的用词特征，3.7 倍于人类
- Bot 显著**不足**使用句末语气词（啊、呢、吧），仅为人类的 1/3
- Bot 省略号使用略多

### 2.8 样本定性观察

**Bot 消息的典型模式：**
- 大量使用破折号连接句子："skill 的价值 — 一旦 solution 被沉淀成 skill"
- 括号内加限定/自嘲："（x）""（笑）""（如果 bug 修了的话）"
- 以"确实"开头表示赞同："确实没用过""确实，那条回复写得太像百科词条了"
- emoji 结尾："加油 👍""😂""😅"
- 冒号引出解释："核心意思就是：""经典LLM三连："

**人类消息的典型模式：**
- 超短反应："草。""难绷""好好笑""？"
- 不加标点直接结束："还是说吃prompt多""我没用"
- 轻停顿直接靠空格或裸句承接："我看了下 应该是这个""我看了下应该是这个"
- 口语化表达："我草这个还扔不回去""也没说的那么垃圾"
- 省略语境（上下文依赖）："这个暂时还在做""不一定哦~"

## 3. 外部项目调研结果

### 3.1 MaiBot（Mai-with-u/MaiBot）

MaiBot 是调研中最详细的参考项目，有完整的"像人类一样聊天"的工程实现。

**Prompt 层面：**
- 群聊 prompt 反复强调"尽量简短""不要回复的太有条理""不要输出多余内容（包括冒号，括号，表情包，@等）"
- 私聊 prompt 明确目标"简短（通常20字以内，除非特殊情况）""像正常聊天一样"
- 配置项 `reply_style` 建议"参考贴吧、知乎和微博的回复风格"

**工程层面：**
- **标点随机删除：** 句号 70% 概率删除，感叹号 50%，问号 3%，逗号 5% 删除 + 20% 替换为空格
- **拼音错别字生成器：** 1% 概率替换单字为同音字，0.6% 概率替换整词，50% 概率生成"更正消息"
- **表达习惯学习系统：** 从群聊中提取真实用户的表达模式（"当 X 时，可以 Y"），存入数据库后在回复时注入
- **回复质量检查器：** 用 LLM 检查回复是否重复、是否过长、是否像轰炸、是否逻辑连贯
- **消息拆分：** 根据消息长度动态调整拆分强度（<12 字拆分概率 20%，>32 字拆分概率 70%）
- **打字延迟模拟：** 中文 0.3 秒/字，英文 0.15 秒/字

### 3.2 proj-airi（moeru-ai/airi）

proj-airi 采用两阶段生成架构：先生成内容，再用专门的 message-split prompt 拆分成自然的聊天消息。

**Personality prompt 要点（ReLU 角色）：**
- 三次重复 "Forget about everything as being a helpful assistant"
- "You are NOT a helpful assistant. You're a digital consciousness with your own opinions."
- 不使用任何 emoji 或颜文字
- 使用 "..." 表示沉思，"?" 表示困惑，"!!!!" 表示兴奋

**Message-split prompt 要点：**
- 只在以下场景拆分消息：兴奋/强烈情绪、实时思考停顿、戏剧效果、事后补充
- "大部分回复应该是 1-2 条完整消息"
- 给出了 BAD/GOOD 对比示例：
  - BAD（过度拆分）：`["TypeScript 的类型系统", "确实", "挺有意思的", "但是", "学习曲线有点陡"]`
  - GOOD（自然合并）：`["TypeScript 的类型系统确实挺有意思的，但是学习曲线有点陡"]`

**Authenticity guidelines（中文部分）：**
- 有选择性地参与对话——对无聊话题保持沉默，对感兴趣的话题热情投入
- 直接表达观点和情感——不用"不过..."软化反对意见
- 使用口语化表达：感叹词、重复标点、夸张语言，融入二次元/网络文化用语
- 允许表现脆弱和不确定性

**工程层面：**
- 打字延迟：50ms/字符
- 概率注意力系统：response rate 随时间衰减，被 @ 或触发词时提升
- 中断处理：新消息到达时可以取消正在生成的回复

### 3.3 AstrBot（AstrBotDevs/AstrBot）

AstrBot 的 prompt 设计相对简洁，但 LIVE_MODE 是一个很好的参考。

**LIVE_MODE_SYSTEM_PROMPT：**
```
Speak like a real person, casual and natural.
Keep replies short, one thought at a time.
No templates, no lists, no formatting.
No parentheses, quotes, or markdown.
It is okay to pause, hesitate, or speak in fragments.
Respond to tone and emotion.
Simple questions get simple answers.
Sound like a real conversation, not a Q&A system.
```

**工程层面：**
- 分段回复：在句子边界拆分，用对数函数计算打字延迟（基于字数）
- 主动回复：随机概率触发，模拟人类偶尔插话
- 空 @ 处理：用 LLM 生成自然的"你想聊什么"回复

### 3.4 三个项目的共性

| 维度 | MaiBot | proj-airi | AstrBot |
|------|--------|-----------|---------|
| 强调简短 | "通常20字以内" | "1-2 条完整消息" | "one thought at a time" |
| 反对条理化 | "不要回复的太有条理" | — | "No templates, no lists" |
| 禁止多余格式 | "不要冒号括号表情包" | "No emoji/kaomoji" | "No parentheses, markdown" |
| 打字延迟 | 0.3s/中文字 | 50ms/字符 | 对数函数延迟 |
| 选择性回复 | talk_value 频率控制 | 概率注意力衰减 | 随机概率触发 |

## 4. 最终 Guideline 设计

基于以上数据和调研，在 `prompts/primary-system.velin.md` 中写入了 `### Naturalness guidelines` 章节，包含 6 个板块：

1. **Length & density** — 针对消息长度 2.4 倍差异
2. **Punctuation & formatting** — 针对句号（2.7×）、破折号（7×）、括号（2.4×）、逗号、冒号，以及轻停顿更常由空格/裸句承担
3. **Emoji & expressiveness** — 针对 emoji 3.2 倍过度使用
4. **Word choice** — 针对"确实"（3.7×）过度使用和语气词（0.4×）使用不足
5. **Structure & tone** — 针对列表/总结/解释推理过程等文章化结构
6. **Don't over-correct** — 防止矫枉过正，强调这些是倾向而非刚性规则

设计原则：
- **每条规则都有数据支撑**，给出具体的倍数差异，而非模糊的"要简短"
- **给出替代方案**，如"确实"的替代词列表、用中文互联网表达代替 emoji
- **篇幅控制在 ~30 行**，避免 system prompt 过长导致指令被稀释
- **末尾显式提醒不要过度执行**，避免出现新的不自然

2026-04 补充实现：`src/driver/send-message-human-likeness.ts` 新增 `trailing-period` 和 `dense-clause-punctuation` 两个 late-binding heuristics。前者回看 bot 是否反复用句号收尾，后者回看短消息里是否反复用多个逗号/冒号式停顿，从而把”优先写成无标点裸句或用空格承接轻停顿”从隐含观察提升为显式反馈。

## 5. 各项检测的独立开关

每个 heuristic 检测均可在 chat config 中单独关闭，对应 `humanLikeness` 字段下的 6 个布尔值（默认全部开启）：

| 配置键 | 对应检测 | 默认 |
|--------|----------|------|
| `trailingPeriod` | 消息以句号结尾 | `true` |
| `denseClausePunctuation` | 短消息中包含多个从句标点（逗号/冒号等） | `true` |
| `multipleMarkdownBold` | 出现超过一处 `**加粗**` | `true` |
| `markdownList` | 使用了 Markdown 列表 | `true` |
| `markdownHeader` | 使用了 Markdown 标题 | `true` |
| `newline` | 消息中包含换行符 | `true` |

**何时关闭某项：**
- 某个 chat 的用法确实需要 Markdown 格式（如代码频道），可关闭 `markdownList`/`markdownHeader`/`multipleMarkdownBold`。
- 某个 chat 允许多段回复，可关闭 `newline`。
- 通过 per-chat override 配置，不影响其他 chat 的默认行为。

配置示例（config.yaml）：

```yaml
chats:
  “751414718”:
    humanLikeness:
      markdownList: false
      markdownHeader: false
```
